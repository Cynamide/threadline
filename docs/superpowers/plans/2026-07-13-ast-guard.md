# AST Guard Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Ship `@threadline/ast-guard` as a local validation package that parses source files, detects boundary violations, and reports stable structured violations.

**Architecture:** Split the package into focused parser, validator, and runner modules. Parsers extract handoffs and import/style facts from source text, validators turn those facts into rule violations, and the runner aggregates file-level results into the shape the CLI can consume.

**Tech Stack:** TypeScript, a real AST parser such as Babel or TypeScript compiler APIs, pnpm workspace, the repo's chosen test runner, and fixture-based tests.

## Global Constraints

- Prefer AST parsing over string matching
- Keep messages actionable
- Make the runner fast enough for local hook use
- Validation should report code, severity, file path, line and column when available, and a message that tells the user what to change
- `HANDOFF001` - missing `id`
- `HANDOFF002` - `id` is not kebab-case
- `HANDOFF003` - missing `title`
- `HANDOFF004` - missing `description` or empty description
- `HANDOFF005` - missing `fallback`
- `STATE001` - `fetch` outside an allowed handoff context
- `STATE002` - `axios` outside an allowed handoff context
- `STATE003` - `useSWR` outside an allowed handoff context
- `STATE004` - `useQuery` outside an allowed handoff context
- `STATE005` - Redux hooks in a UI component
- `STATE006` - localStorage or sessionStorage access in a UI component
- `STATE007` - router navigation in a UI component
- `STYLE001` - global CSS in a Tailwind project
- `STYLE002` - class name does not fit the configured styling strategy
- `STYLE003` - non-module CSS file in CSS modules mode
- `PATH001` - modified `src/api/`
- `PATH002` - modified `src/store/`

- [ ] **Step 3: Implement the parser and validator shells**

```ts
export function parseHandoffs(sourceCode: string, filePath: string): ParsedHandoff[] {
  const ast = parse(sourceCode, { sourceType: 'module', plugins: ['typescript', 'jsx'] });
  return collectHandoffCalls(ast, filePath);
}

export function runValidation(options: ValidationOptions): ValidationResult {
  const handoffs = options.files.flatMap((file) => parseHandoffs(file.source, file.path));
  const violations = handoffs.flatMap(validateHandoffSyntax);

  return {
    passed: violations.length === 0,
    summary: {
      filesValidated: options.files.length,
      handoffsFound: handoffs.length,
      errorCount: violations.filter((violation) => violation.severity === 'error').length,
      warningCount: violations.filter((violation) => violation.severity === 'warning').length
    },
    violations
  };
}
```

  return findCallExpressions(ast, 'handoff')

- [ ] **Step 3: Implement field-level validation and message formatting**

```ts
export function validateHandoffSyntax(handoff: ParsedHandoff): Violation[] {
  const violations: Violation[] = [];
  if (!handoff.id) violations.push({ code: 'HANDOFF001', severity: 'error', file: handoff.filePath, message: 'Add a stable string literal `id`.', line: handoff.line, column: handoff.column });
  if (handoff.id && !/^[a-z][a-z0-9]*(?:-[a-z0-9]+)*$/.test(handoff.id)) violations.push({ code: 'HANDOFF002', severity: 'error', file: handoff.filePath, message: 'Use kebab-case for `id`.', line: handoff.line, column: handoff.column });
  if (!handoff.title) violations.push({ code: 'HANDOFF003', severity: 'error', file: handoff.filePath, message: 'Add a human-readable `title`.', line: handoff.line, column: handoff.column });
  if (!handoff.description?.trim()) violations.push({ code: 'HANDOFF004', severity: 'warning', file: handoff.filePath, message: 'Add a `description` when the implementation needs context.', line: handoff.line, column: handoff.column });
  if (!handoff.fallback) violations.push({ code: 'HANDOFF005', severity: 'error', file: handoff.filePath, message: 'Provide a callable `fallback`.', line: handoff.line, column: handoff.column });
  return violations;
}
```

- [ ] **Step 3: Implement styling-strategy checks for Tailwind and CSS modules**

```ts
export function validateStylingScope(source: string, filePath: string, strategy: StylingStrategy): Violation[] {
  if (strategy === 'tailwind' && /\.css$/.test(filePath)) {
    return [makeViolation('STYLE001', filePath, 'Move component styling out of global CSS in Tailwind projects.')];
  }
  if (strategy === 'css-modules' && /\.css$/.test(filePath) && !/\.module\.css$/.test(filePath)) {
    return [makeViolation('STYLE003', filePath, 'Use CSS module files for component styling.')];
  }
  return [];
}
```
